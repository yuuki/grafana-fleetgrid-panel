# ClusterView パネルプラグイン設計仕様書

日付: 2026-07-18
状態: 承認済み設計の記録(実装計画は別文書)

## 概要

AI/HPCクラスタ監視のためのGrafanaパネルプラグイン **ClusterView** を新規開発する。
zone、ノード、GPUのような入れ子の物理構成を階層グリッドとして描画し、Prometheus / VictoriaMetricsのメトリクスをセルの色として俯瞰できるようにする。
既存の hpe-grafana-clusterview-panel の代替であり、同プラグインの運用で確認された6つの課題を解決する。

## 背景

既存プラグイン(HewlettPackard/hpe-grafana-clusterview-panel 1.3.2)には次の課題がある。

1. **色分けの手間**:色は条件式(Conditions)の手動設定のみで、値に応じた連続グラデーションがない。
2. **数値の視認性**:セルは固定ピクセル寸法で、数値が潰れてもそのまま描画される。
3. **自動ソート**:ラベル中の数値を認識したソートがない。
4. **単位表現**:数値+単位のフォーマット(Gbpsなど)に対応しない。
5. **即時ドリルダウン**:セルクリックはURLテンプレートによる遷移のみで、その場で時系列を確認できない。
6. **複数メトリクスの比較困難**:1パネル1メトリクスのため、メトリクス数が増えるとパネル間の見比べが必要になる。

また実装面では、Grafana 9.5 APIとReact 17に依存し、階層が8段固定で、色と単位が独自実装になっている。
これらは現行のGrafanaプラグイン規約に沿ってゼロから設計し直す。

## 対象環境と制約

- データソースは Prometheus / VictoriaMetrics に限定する。
- Grafana 11.6.0 以降を対象とする(`grafanaDependency: >=11.6.0`)。12以降の機能が必要になった場合は下限を引き上げてよい。
- `@grafana/create-plugin` の標準構成(TypeScript、React 18、webpack/SWC、Jest、Playwright)に準拠する。

## 確定要件

| # | 課題 | 決定方針 |
|---|------|---------|
| 1 | 色分け | Grafana標準のColor scheme(連続グラデーション)とThresholdsに全面移行する。色分け対象は数値のみ |
| 2 | 数値視認性 | 色による俯瞰を優先する。セルサイズは自動計算し、テキストが収まるときだけ数値を描画する。ホバーで常に値を確認できる |
| 3 | 自動ソート | natural sortを既定とする(昇順、降順、無効を選択可) |
| 4 | 単位 | Grafana標準のUnit設定を使う |
| 5 | ドリルダウン | セルクリックでポップオーバーに時系列を表示する。Data Links設定済みのフィールドではリンク遷移を優先する |
| 6 | 複数メトリクス | 単一メトリクス表示+パネル内切替を既定とし、分割セル表示をオプトインで提供する。ツールチップとポップオーバーにはデータを返した各メトリクス(metricInfos基準)を並べる |
| - | 階層定義 | ラベル自動検出+選択式エディタ+ライブプレビュー。層数は可変 |
| - | 欠損ノード | 全クエリのunionでノード集合を構成し、値のないセルを欠損色で表示する。全ノード集合が必要な場合は `up` 系などのクエリを追加して表現する |

## 全体アーキテクチャ

### 描画方式

単一の `<canvas>` に全セルを描画し、ツールチップ、ポップオーバー、凡例、メトリクスセレクタをDOM(React)で重ねる。

DOM方式(既存プラグインと同じ)を採らない理由は規模にある。
想定規模は約300ノード×8 GPU≒2,400セルで、分割セル表示では最大15,000区画に達する。
この数のdiv要素はReactの再レンダリングとブラウザのレイアウト計算の両方で引っかかりを生む。
Canvasであれば区画数が増えても1パスの描画で済み、数値やラベルのテキストも同時に描ける。

ホバーとクリックの対象セルは、マウス座標からの逆引き(ヒットテスト)で特定する。
レイアウトを自前で計算しているため、逆引きは算術的に行える。

トレードオフとして、スクリーンリーダー対応はDOM方式より弱い。
監視ダッシュボードという用途を踏まえ、許容する。

### モジュール構成

```
src/
  module.ts           パネル登録とオプション定義
  types.ts            オプション型とセルモデル型
  data/               DataFrame群からセルモデルへの変換(純関数)
    hierarchy.ts        ラベル抽出、正規表現、natural sort、ツリー構築
    values.ts           複数クエリのunion、値解決、欠損判定
  layout/             レイアウトエンジン(純関数)
    layout.ts           ツリーとパネル寸法から座標と区画を計算
  render/             Canvas描画とヒットテスト
  components/         Reactコンポーネント(パネル本体、ポップオーバー、凡例)
  options/            階層定義エディタ(カスタムエディタ)
```

データ変換とレイアウト計算を描画から分離した純関数として実装する。
数千セル規模のロジックをJestで直接検証でき、描画層を薄く保てるからである。

## データパイプライン

### 入力データ契約

Prometheus / VictoriaMetrics の2つのクエリ形式を受け付ける。

- **Time series形式**(既定):各系列が1フレームで、`field.labels` にラベルを持つ。
- **Table形式**(instantクエリ):1フレームでラベルが列に展開される。

クエリはinstantを推奨する。
表示に必要なのは現在値のみであり、転送量が最小になるからである。
rangeクエリが与えられた場合は、系列ごとに時間方向のreduce(既定は最後の非null値)で現在値に畳む。
reduceの計算方法は、数値を返すCalculationに限定したセレクタで変更できる(配列やbooleanを返すCalculationはセル値の契約を破るため選択肢に載せない)。

### ステージ構成

```
PanelData (Grafanaから受領)              パネルオプション / Field設定
        │
        ▼
[1] normalize()      形式差の吸収 → 行リスト {labels, value, refId}
        ▼
[2] buildHierarchy() 階層キー抽出 + natural sort → ツリー
        ▼
[3] buildCells()     全クエリunion + 欠損判定 + 空間集約 → セルモデル
        ▼
[4] resolveDisplay() Grafana標準display processorで値から色と整形テキストを解決
        ▼
[5] computeLayout()  ツリーとパネル寸法から座標を計算
        ▼
[6] renderCanvas()   描画
```

### 各ステージの入出力

クエリA `round(DCGM_FI_DEV_POWER_USAGE{job="dcgm-exporter"}, 1)`(電力)とクエリB(温度)の2クエリを例に示す。

normalize() はフレーム形式の違いを吸収し、1行が `{labels, value, refId}` のフラットな行リストを返す。

| refId | zone | host.name | gpu | value |
|-------|------|-----------|-----|-------|
| A | zone-a | node-a017 | 0 | 503 |
| A | zone-a | node-a017 | 1 | 480 |
| B | zone-a | node-a017 | 0 | 61 |
| B | zone-a | node-a040 | 0 | 28 |

buildHierarchy() は階層定義(後述)を各行に適用し、階層キーを持つツリーを構築する。
枝はレベルごとの設定に従いnatural sortする。

buildCells() は葉ごとに全クエリの値を束ねる。

```
cell("zone-a/017/0") = { A: 503,  B: 61 }
cell("zone-a/040/0") = { A: null, B: 28 }
```

セルは全クエリのunionから生成するため、一部のクエリにしか存在しないノードもツリーに現れる。
あるクエリに該当系列がないセルは、そのクエリの表示時に**欠損**として欠損色で描画する。
同一の階層キーに複数系列が落ちた場合(階層をノード単位までにして8 GPU分が1セルに畳まれる場合など)は、**空間集約**関数で1値にする。
集約関数はオプションで `max`(既定)、`mean`、`min`、`sum` から選ぶ。
既定を `max` とするのは、異常検知の用途では最悪値の表示が安全側だからである。

resolveDisplay() は各セルの色と表示テキストを解決する。
色と整形は自前で計算せず、`@grafana/data` の display processor(fieldのColor scheme、Thresholds、Unit、Decimals、Min-Max設定を解釈する標準機構)に委譲する。

### スケールの独立性

min/maxが明示設定されていないクエリは、そのクエリ(refId)内のデータ範囲からmin/maxを自動計算する。
電力(600〜1000 W)と温度(30〜90 ℃)が同じスケールに潰されることを防ぐためである。
固定したい場合はGrafana標準のOverrides(by query)で設定する。

### エラー処理

- 階層定義に指定したラベルがデータに存在しない場合、「ラベル `xxx` がクエリ結果にありません(検出されたラベル: ...)」をパネルに表示する。
- 正規表現が全行にマッチしない場合も同様に警告を表示する。黙って空表示にしない。
- データが空の場合はGrafana標準の「No data」を表示する。

## 階層定義

### レベル定義エディタ

パネルエディタでクエリ結果に含まれるラベルキーを自動列挙し、ドロップダウンから選んで並べる。
各レベルには検出されたグループ数とサンプル値を即時表示し、設定ミス(マッチせず全欠落など)がその場で見えるようにする。

```
階層レベル(並べ替え可)
1. ラベル: [zone ▾]       → 3グループ: zone-a, zone-b, zone-c
   抽出: [そのまま ▾]      レイアウト: [縦積み ▾]
2. ラベル: [host.name ▾]  → 312グループ: 004, 017, 018, …
   抽出: [末尾の数値 ▾]    レイアウト: [グリッド ▾] 列数 [20]
3. ラベル: [gpu ▾]        → 8グループ: 0, 1, …, 7
   抽出: [そのまま ▾]      レイアウト: [グリッド ▾] 列数 [2]
[+ レベル追加]
```

レベル数は可変で、上限を設けない(実用上は8段程度を想定)。

### 抽出プリセット

ラベル値から階層キーを切り出す方法は3択のプリセットとする。

- **そのまま**:ラベル値を階層キーとして使う。
- **末尾の数値**:`node-a004` から `004` を切り出す(`/(\d+)$/` 相当)。
- **カスタム正規表現**:第1キャプチャグループを階層キーとする。既存プラグインの `node-.+(\d\d\d)` のような指定をそのまま移行できる。

### ソート

レベルごとにnatural sort(昇順)を既定とし、降順と無効(データ出現順)を選べる。
natural sortは文字列中の数値部分を数値として比較する(`002 < 010`、`node-a2 < node-a10`)。

## レイアウトエンジン

### レベルレイアウト

各レベルに4方式のレイアウトを指定できる。

- **縦積み**(vertical):子要素を縦に並べる。
- **横並び**(horizontal):子要素を横に並べる。
- **折返し**(flow):横に並べ、幅を超えたら折り返す。
- **グリッド**(grid):指定列数で格子状に並べる。

レベルごとに枠線とグループラベルの表示を切り替えられる。

### セルサイズの自動計算

既存プラグインの固定ピクセル指定を廃止し、パネル寸法から自動計算する。

葉セルの一辺 `s` を大きくするとツリー全体の必要サイズはおおむね増えるが、折返しレイアウトの折返し数が変わる境界では逆転が起こりうる(`s` に対して単調ではない)。
このため二分探索は使わず、上限から0.5px刻みで `s` を下げながら最初にパネルへ収まる値を採用する(最大69候補。ツリー走査は数千セルでもミリ秒オーダーで、走査全体も実用上問題ない)。

- `s` が下限(6px。色の判別が保てる最小値)を下回る場合は下限に固定し、パネル内の縦スクロールに切り替える。スクロール時はグループラベルを上端に固定表示する。
- `s` が上限(40px)を超える場合は上限で止める。

グループ間余白、ラベル行高さ、枠線幅は計算に含める。
これらはテーマ定数として固定し、オプションにしない。

### 数値表示の判定

「余裕があるときだけ数値を描画する」を次の規則で機械的に判定する。

1. フォントサイズをセル高さから決める(`s × 0.38` を9〜15pxにclamp)。
2. Canvasの `measureText` で整形済みテキスト(例 `"503 W"`)の描画幅を実測する。
3. `テキスト幅 + 4px ≤ セル幅` のときだけ描画する。
4. 単位付きで収まらない場合は数値のみ(`"503"`)で再判定し、それでも収まらなければ描画しない。

テキスト色は背景色の輝度から白/黒を自動選択する(Grafana標準のcontrastユーティリティを使う)。
分割セルモードの区画には数値を描画しない。
区画は必然的に小さく、判読可能になるケースがないからである。

## 表示モードと複数メトリクス比較

セルモデルは常に全クエリの値を保持し、描画モードだけを切り替える。

### 単一モード(既定)

セルは選択中の1メトリクスの色で全面を塗る。
パネル上部のセレクタで表示メトリクスを1クリックで切り替えられる。
セレクタの選択はビューアのローカル状態であり、ダッシュボードの変更扱いにならない。
初期選択はオプション「既定の表示メトリクス」(未指定なら先頭クエリ)。

### 分割セルモード(オプトイン)

パネルオプションで有効化した場合のみ、データを返したクエリ数(metricInfos基準、最大9件)に応じてセルを自動分割し、それらを同時に表示する。

- 区画割り: 2クエリ=左右分割、3=縦3分割、4=2×2、5〜6=3×2、7〜9=3×3。
- 区画は最大9とする。10クエリ以上ある場合は先頭9クエリのみを区画に割り当て、凡例に対象外のクエリがある旨を表示する。
- 区画順はクエリ(refId)順で左上から右へ。
- 各区画の色はそのクエリ自身のスケールに対するグラデーション。
- パネル上部の凡例に区画の位置対応図とクエリ名を表示し、帰属を明示する。

分割セルを既定にしない理由は、Grafana標準パネルにない表現であり、区画の色の帰属を凡例と突き合わせる認知負荷があるからである。
常時の俯瞰は単一モードで行い、同時比較が必要なダッシュボードだけが分割を有効化する。

## 色と単位

色、単位、桁数、min/max、Data LinksはすべてGrafana標準のFieldタブとOverridesタブで設定する。
パネル独自の色設定は欠損色のみとする。

- 連続グラデーション: Color schemeで `Green-Yellow-Red (by value)` などを選ぶ。
- 閾値による離散色: 標準のThresholdsを使う。
- クエリごとの設定: override(by query)で行う。

既存プラグインのConditions式(独自DSL)は実装しない。
色分け対象は数値のみである。

## ドリルダウンとインタラクション

### ホバー: ツールチップ

マウス座標のヒットテストでセルを特定し、DOMツールチップに階層パスと全メトリクスの現在値を表示する。

```
zone-a / 017 / GPU 3
● 電力    621 W
● 温度     78 °C
● 使用率   97 %      (●は各メトリクスのその値の色)
```

### クリック: Data Links優先、なければポップオーバー

- フィールドにGrafana標準のData Linksが設定されている場合はリンク遷移する(複数リンクはパネル独自のリンク選択メニューを表示し、テーマ配色と可視範囲クランプを適用する)。
- Data Linksがない場合はポップオーバーを表示する。

### ポップオーバー

クリックしたセルの近傍にカード型で表示する。
パネル端でははみ出さない側に自動配置し、外側クリックまたはEscで閉じる。
開いたまま別セルをクリックすると内容が切り替わる。

内容は階層パスと、メトリクスごとの現在値の行である。時系列が取得できた場合のみ各行にスパークラインを添える。
1チャートに重ねない理由は、メトリクス間で単位もスケールも異なるからである。
時間範囲はダッシュボードと同期する。

### 時系列データの取得

- パネルのクエリがrangeの場合、受信済みフレームに時系列があるため再クエリしない。
- instantの場合、クリック時に `getDataSourceSrv()` 経由でそのパネルの各クエリをダッシュボードの時間範囲でrange再実行し、クリックセルのラベルに一致する系列だけを抽出して描画する。`maxDataPoints` は約100点に制限し、結果はパネル単位で保持して次のデータ更新まで再利用する(1回の再クエリで全系列を取得するため、別のセルを続けて開いても再取得は発生しない)。

instant時の再クエリは全系列を取得してクライアント側で1系列を選ぶ形になる。
`round()` などの関数適用でメトリクス名ラベルが失われるため、クエリ式への系列セレクタの自動注入は一般には安全にできないからである。
再クエリが遅いと感じる規模では、rangeクエリ運用に切り替えれば再クエリ自体がなくなる。

## パネルオプション体系

Grafana標準機構で設定できるものはパネル独自オプションにしない。

| カテゴリ | オプション |
|---------|-----------|
| Hierarchy | レベルリスト(ラベル、抽出プリセット、ソート、レイアウト方式、グリッド列数、枠線、グループラベル表示) |
| Display | 表示モード(単一/分割)、既定の表示メトリクス、数値表示(auto/off)、欠損色 |
| Data | 空間集約(max/mean/min/sum)、時間方向reduce(数値Calculationに限定したセレクタ) |

## テスト戦略

| 層 | 対象 | 手段 |
|----|------|------|
| 単体(Jest) | normalize、buildHierarchy(抽出とnatural sort)、buildCells(unionと欠損と空間集約)、computeLayout(自動セルサイズとスクロール切替と分割区画)、数値表示判定 | 実データ形状を模したフィクスチャで純関数を直接検証する |
| コンポーネント(React Testing Library) | ツールチップとポップオーバーの開閉、メトリクスセレクタ、階層定義エディタのラベル列挙 | jsdomとcanvasモック |
| E2E(Playwright、`@grafana/plugin-e2e`) | ダッシュボード上での描画、メトリクス切替、クリック動作 | create-plugin標準のprovisioningとdocker-compose環境 |
| 手動確認 | グラデーションの見え方、数値の可読性、ポップオーバー位置 | 開発サーバーで実施し、スクリーンショット付きで報告する |

## スコープ外

次の項目は要件にないため実装しない。

- 文字列値への色分け(Value mappingsによる色付けを含む)
- 修飾キーによるクリック挙動の切替(Cmd+クリックなど)
- ポップオーバーのピン留め、セル選択状態の保持
- 階層の自動推論(ラベルの包含関係からの自動配置)
- ズーム、パン操作
- 既存プラグインのConditions式とURLテンプレートの互換実装
